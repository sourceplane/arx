## Objective

Implement the first coherent Plan Studio slice: make `LiveOrunService.GeneratePlan` call Orun planning internals directly, wire Plan Studio form/generate/review/error state through Bubble Tea messages, render a useful textual DAG review, and add focused unit/property tests for the Plan Studio state machine.

This satisfies `.kiro/specs/orun-tui-cockpit/requirements.md` Requirement 4 acceptance criteria 4.1-4.4 and the service-boundary portions of Requirement 13. It intentionally stops before dry-run/run execution because `RunPlan` is still a Phase 3 stub.

## PR Boundary

1. Implement `internal/tui/services/plan_service.go` (or equivalent) so `LiveOrunService.GeneratePlan(ctx, services.PlanRequest)` loads/resolves the intent, resolves compositions/presets, normalizes/validates, expands instances, filters by environment/components/changed scope as supported by the service request, plans jobs, resolves promotion dependencies, topologically sorts, renders a `*model.Plan`, computes/stores checksum metadata, and returns `services.PlanResult`.
2. Replace the placeholder `internal/tui/views/plan_studio.go` with a focused Plan Studio model: scope/environment/trigger/base/head/dry-run form state, `PhaseFormFill`, `PhaseGenerating`, and `PhaseReview`, async `GeneratePlan` command, error recovery to form-fill, and a textual DAG/review rendering with job IDs, dependencies, profile/gate/destructive-capability hints, composition source digest/count, checksum, warnings, and component list.
3. Wire the root TUI model enough for Plan Studio entry and plan-generation messages: Browse `g`/`p` or command-palette `:plan changed` may switch to Plan Studio if already easy, but the minimum required entry is a tested key path from the root model into `ModePlanStudio` and handling of `services.PlanGeneratedMsg` so errors show through the existing error banner.
4. Add focused tests for `GeneratePlan` and Plan Studio state transitions, including property/unit coverage for: successful `PlanGeneratedMsg` enters review, errored `PlanGeneratedMsg` returns to form-fill with an error, `esc` returns to form-fill, and no direct execution/run transition is possible in this slice.
