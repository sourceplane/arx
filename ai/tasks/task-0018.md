## Objective

Rewire `orun run` to the canonical revision-first flow described in
`specs/orun-state-redesign/cli-surface.md` §2 and
`compatibility-and-migration.md` §3. Concretely:

1. Resolve a `PlanRevision` via `internal/revision.ResolveRevision` using the
   canonical 7-step chain in compat-and-migration.md §3:
   1. arg empty → `refs/latest-revision.json`.
   2. arg matches an existing file → load plan from file; synthesize
      `system.manual` revision **in-memory** (no disk write).
   3. arg matches the revision-key regex → `revisions/<arg>/plan.json`.
   4. arg matches a named ref → `refs/named/<arg>.json` → revision key.
   5. arg is hex (legacy plan hash) → `.orun/plans/<arg>.json` (legacy load
      + synthesize a migrated revision in-memory; `triggerType: "system"`,
      `triggerName: "system.migrated"`, `mode: "migration"`).
   6. arg matches a component name → existing component-run behavior
      (unchanged).
   7. otherwise → typed error "ambiguous or unknown run target".
2. Add `--revision <key>` flag that **skips the resolution chain** and loads
   `revisions/<key>/plan.json` directly. Existing flags (`--plan`,
   `--exec-id`, `--dry-run`, `--runner`, …) unchanged.
3. Create the execution via `internal/executionstate.CreateExecution`:
   - derive `executionKey` (`run-NNN` for the canonical case;
     `statestore.SanitizeExecID(<value>)` when `--exec-id <value>` is passed,
     preserving the original in `OriginalKey` per
     compat-and-migration.md §1).
   - write `revisions/<revKey>/executions/<execKey>/{execution,
     snapshot.latest}.json`.
   - write `indexes/executions/<execKey>.json`.
   - write `refs/latest-execution.json`.
4. Hook the runner snapshot stream into
   `internal/executionstate.Bridge.MirrorRunnerOutput`. The bridge is invoked
   on each runner tick that produces or updates the legacy
   `.orun/executions/<legacyExecID>/state.json` /
   `metadata.json` so the new layout converges in real time. **Failures
   return nil** per the M4 contract — bridge errors must never crash the
   run; rely on the resolver legacy-fallback (M4 PR-A) for convergence.
5. On terminal status: update `execution.json`, `refs/latest-execution.json`,
   and the manifest summary via
   `internal/executionstate.UpdateLatestExecutionSummary`.
6. Pin the `bridge-mirror-failed` payload schema in `data-model.md` §9 with
   the exact field set already locked in code (Task 0014):
   `{executionKey, revisionKey, legacyExecId, artifact, stage, mode, error}`
   and `stage ∈ {read-source, read-dest, translate-dest, mkdir-dest,
   remove-dest, link, copy}`. This is a strictly normative pin of an existing
   shape, not a behavior change.

## Required Outcomes

1. **All 7 resolution branches wired and tested.** A table-driven test in
   `cmd/orun/` covers each numbered branch in compat-and-migration.md §3
   plus the `--revision <key>` direct path. Branch 2 (file) and branch 5
   (legacy hex) materialize a synthesized revision **in-memory only** — no
   disk write, asserted by inspecting the workspace after the run.
2. **`metadata.trigger` + `metadata.revision` flow through to the
   execution.** `CreateExecutionInput` carries the resolved
   `TriggerOccurrence` and `PlanRevision`; `execution.json` records the
   revision key and trigger reference. The synthesized in-memory cases set
   `triggerType: "system"`, `triggerName: "system.migrated"` (branch 5) or
   `triggerName: "system.manual"` (branch 2), per
   compat-and-migration.md §3 and §4.
3. **Bridge mirroring runs on every snapshot write.**
   `Bridge.MirrorRunnerOutput(ctx, execKey, revKey, legacyExecID)` is called
   after each runner tick that updates the legacy execution dir. The bridge
   uses `MirrorModeAuto` (zero value) so hardlink-first/copy-on-EXDEV is the
   local-FS behavior. Bridge errors are logged but never returned (M4
   contract: failures return nil).
4. **`--revision <key>` skips the chain.** When set, the resolver is called
   with the canonical key form only; arg-parsing for `latest`/file/hex/named
   is bypassed. Mutually exclusive with positional `<arg>`; surface a clear
   error if both are given.
5. **`--exec-id <value>` preserves OriginalKey.** When passed,
   `executionKey = statestore.SanitizeExecID(value)` and the original is
   retained in `ExecutionRun.OriginalKey`. Asserted by an integration test
   that uses a value containing chars that must be sanitized (`gh-12345-1-abc`
   passes through unchanged; `weird name/with slash` becomes
   `weird-name-with-slash` with `OriginalKey = "weird name/with slash"`).
6. **Terminal-state summary update.** On the last terminal status, call
   `internal/executionstate.UpdateLatestExecutionSummary` so
   `refs/latest-execution.json` and the manifest summary block reflect the
   final outcome. Verified by reading the refs after the test run.
7. **Legacy compat preserved.** `orun run --plan <hash>` still works against
   a pre-revision-era workspace (only `.orun/plans/<hash>.json` exists) via
   branch 5 of the resolution chain. `orun run` against a
   post-M5.a workspace finds `refs/latest-revision.json` and uses it
   (branch 1).
8. **`data-model.md` §9 pins the bridge-mirror-failed schema.** Schema text
   matches the field set already in code: `{executionKey, revisionKey,
   legacyExecId, artifact, stage, mode, error}` with the seven `stage`
   values. Note that this is a normative pin of existing code, not a
   behavior change.
9. **New on-success summary block.** Print a `cli-surface.md` §2 summary
   block in the same shape M5.a established (prefix-then-detail-line: a
   §2 header followed by the existing legacy detail line so existing
   tooling that scans the legacy line keeps working). Suggested fields:

## Risk Notes Carried Forward

These are pre-existing risks the verifier will check do not regress in M5.b:

- **`MirrorRunnerOutput` becomes load-bearing in this task** — it's been
  exercised through tests only since Task 0014. M5.b is the first
  production caller. Prefer-revision-first-with-legacy-fallback resolver
  (M4 PR-A) means a missed mirror only delays convergence, not correctness;
  failures must continue to return nil per the M4 contract.
- **`MirrorMode = MirrorModeAuto`** is the right default for the local
  driver (zero value; hardlink-first, copy-on-EXDEV). Phase 2 remote
  drivers (S3/GCS) would set `MirrorModeCopy` explicitly; do not anticipate
  that here.
- **`emitFailure` is best-effort** — events-dir-unwritable failures are
  silently dropped. M5.b should add stderr/metric fallback for the bridge's
  emitFailure path. If you don't ship a fallback in this PR, record the
  decision in the report so M5.c/d can pick it up.
- **Event-sequence retry budget of 32** is acceptable for single-writer
  Phase 1; do not change it.
- **`MirrorModeHardlink` debug-fold decision deferred to M6** — do not
  remove it, do not advertise it in `--help`, do not add new callers. It
  exists for drift-detection tests only.
- **`internal/executionstate` coverage at exact 90.0 % floor** — small
  refactors that delete a covered branch could trip the gate. Add tests for
  any new code paths you introduce in `internal/executionstate` (which
  should be minimal — the writer/bridge are already complete).
- **Half-shipped delivery anti-pattern** — explicit `gh pr list --head`
  check before reporting PR number; never report TBD.
