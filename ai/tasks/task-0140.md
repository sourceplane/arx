# Task 0140

Agent: Implementer

## Current Repo Context

- PR #139 (`fix: enable built-in artifact upload in GitHub Actions`) has been merged into `main` as merge commit `06bd8c8`; main-branch CI and remote-state conformance runs for the merge both succeeded.
- The active roadmap remains GitHub Artifacts stabilization. The GitHub Artifacts spec lists high-priority gaps in `.kiro/specs/github-artifacts/requirements.md` and `.kiro/specs/github-artifacts/design.md`.
- Current code reality on `origin/main` has `orun github logs` downloading shards but printing only `manifest.Files` logical keys instead of actual log contents, and it lacks tests that prove log content is emitted.
- TUI cockpit specs exist under `.kiro/specs/orun-tui-cockpit/`, but that work should wait until the current GitHub Artifacts operator-facing gaps are closed.

## Objective

Implement `orun github logs` content display so local users can read CI job/step logs directly from GitHub artifact shards without first running `orun github pull`. This satisfies GitHub Artifacts Requirement 14 and adds the missing CLI test coverage from Requirement 20 for the log-content path.

## PR Boundary

One PR should include:

1. Update `cmd/orun/command_github.go` so `runGithubLogs()` prints actual log file contents from downloaded job shards.
2. Add helper code as needed to safely resolve `log:*` manifest entries under the extracted shard directory and warn/continue when one log file cannot be read.
3. Add focused tests in `cmd/orun/command_github_test.go` or nearby test files proving log content, per-step headers, job filtering/no-match behavior, and unreadable-log warning behavior.
4. If needed for testability, introduce small seams or helpers; keep them local to the GitHub CLI/log path.

Explicit non-goals:

- Do not implement `orun github runs --details` / Level 2 manifest downloads.
- Do not implement `orun github pull` hydration changes, partial status display, or E2E CI workflow tests.
- Do not start `orun tui` cockpit implementation.
- Do not rewrite the GitHub artifact client or shard schema.
- Do not change artifact naming, upload, retention, or CI workflow behavior.

## Required Outcomes

- [ ] `orun github logs` downloads matching job shards and prints each actual log file's content to stdout.
- [ ] Each printed log section includes a header in the format `=== {shard-name} / {step-id} ===`.
- [ ] Only manifest entries whose logical name starts with `log:` are printed as log content; non-log files are ignored.
- [ ] The implementation resolves log file paths under the extracted shard directory and does not weaken existing path traversal defenses.
- [ ] If a log file cannot be read, the command prints a warning to stderr and continues with remaining log files/shards.
- [ ] `--job` filtering remains supported and still errors when no artifact name matches the filter.
- [ ] Focused tests cover the new behavior without requiring live GitHub API access.
- [ ] Existing GitHub/runbundle/artifactstore tests still pass.
- [ ] Implementer report is written to `ai/reports/task-0140-implementer.md` and contains the real PR number, not `TBD`.

- No new GitHub Actions workflows or live E2E tests in this PR.
- No changes to artifact upload helper, runtime token export, or `orun plan/run --artifact github` behavior.
- No TUI cockpit code or dependency changes.
- No broad cleanup, formatting churn, or unrelated refactors.

## Constraints

1. Follow the existing shard trust model: downloaded shards are extracted by `Client.Download()` and validated by `runbundle.ReadShardManifest`; do not bypass or weaken validation.
2. Treat log contents as user-visible CI output. Do not print tokens or credentials beyond what is already present in stored logs; do not add any new token/env dumping.
3. If joining `ds.Dir` and a manifest relative path, ensure the resolved path remains inside `ds.Dir` before reading, even though the downloader and reader already perform traversal checks.
4. Preserve the existing warn-and-continue behavior for individual shard download failures.
5. Keep the PR focused on Requirement 14 and its tests. If `runs --details` looks adjacent, leave it for a later task.
6. Create/reuse a task branch, commit task-scoped changes, push, and open a GitHub PR before reporting complete.
7. After `gh pr create`, update the implementer report with the actual PR number and push that report update. `PR Number: TBD` is not acceptable.
