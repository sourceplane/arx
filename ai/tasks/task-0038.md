# Objective

Ship the remaining C5 catalog read commands in `cmd/orun/`, each reusing
the shared selector parser, the §11 envelope writer, and the Resolver:

1. **`orun catalog list`** (`catalog_list.go`) — enumerate the components in
   the selected catalog. Default columns `COMPONENT, TYPE, OWNER, SYSTEM,
   LAST EXEC, STATUS` (`cli-surface.md` §3). Filter flags `--owner`,
   `--system`, `--domain`, `--type`, `--status`. `--json` emits one object
   per row with keys `{componentKey, name, type, owner, system,
   lastRevisionKey, lastExecutionKey, lastExecutionStatus,
   sourceSnapshotKey, catalogSnapshotKey}`. Resolve the catalog via the
   shared selector (default `current`), enumerate its components, derive
   `type`/`owner`/`system` from each resolved `ComponentManifest`, and the
   `lastExecution*` / `STATUS` columns from `ResolveComponentLatest`. Sort
   by `componentKey` for byte-stable output.

2. **`orun catalog describe <component>`** (`catalog_describe.go`) — resolve
   one component selector to its `ComponentManifest`. Text sections in the
   §4 order (Component, Ownership, Source, Environments, Profiles,
   Dependencies, APIs, Resources, Runtime inference, Latest executions,
   Resolution provenance). `--json` emits `{manifest, executions: []}`
   (manifest + the catalog-local `ComponentExecutionIndex` rows). Selector
   rules (§4): bare name resolved within the current repo; fully-qualified
   key exact match; **ambiguous bare name across repos → exit 4 with the
   candidate list**.

3. **`orun catalog tree`** (`catalog_tree.go`) — render the catalog graphs.
   Optional positional `<component>` roots the walk; `--direction` ∈
   `out` (default) | `in` | `both`. Text output is a left-aligned tree with
   `→` arrows + edge-type annotations. `--json` emits `{nodes, edges}`
   matching the `CatalogGraph` shape. Walk the resolved
   `CatalogSnapshot`/`CatalogGraph`s — do not recompute graphs in the CLI.

4. **`orun catalog history <component>`** (`catalog_history.go`) — read-only
   enumeration of the component execution history from the catalog-local
   `ComponentExecutionIndex`. Default columns `TIME, REVISION, EXEC,
   TRIGGER, PROFILE, ENV, STATUS`, newest-first, default `--limit 50`.
   Filter flags `--trigger`, `--profile`, `--environment`. `--json`
   supported. **Read-only: no event APPEND (that is C7).** History here is
   enumeration over the already-persisted execution index, not the
   `ComponentCatalogEvent` log.

5. **`orun catalog validate`** (`catalog_validate.go`) — re-run resolution
   in strict mode against the selected source and report the typed
   `[]ValidationIssue` from `resolution-pipeline.md` §6 + §8. Exit 1 on any
   error; exit 0 with warnings unless `--strict` (then any warning → exit
   1). `--source` selector supported. `--json` supported. **`--rebuild-
   indexes` is C8 — do NOT wire it here** (the resolver method exists but
   this flag is out of scope).

6. **`orun catalog diff`** — **STUB ONLY** (`catalog_diff.go`). Register the
   cobra command so it appears in the root help index, but its `RunE`
   prints a one-line `catalog diff is not yet implemented (C8)` and exits a
   documented non-zero (suggest exit 2 / internal-not-ready, your call —
   justify in report). No flags beyond what the help index needs. Full impl
   is C8.

Plus: register all six on the `orun catalog` root (extend
`registerCatalogCommand`), update the root subcommand-index `Long` help to
list them, and ship help-text fixtures for every new command under
`cmd/orun/testdata/help/catalog/*.txt`.

# Integration Notes

The crux design decision is the **`list`/`describe` data source**.
`cli-surface.md` §3 wants `OWNER`/`SYSTEM`/`TYPE`/`STATUS` columns and
`--owner`/`--system`/`--domain`/`--type` filters. PR-1 left the non-
component axes of `CatalogLocalIndexes` (`Owners`/`Systems`/`Domains`/
`Types`) **empty** (only the component execution axis is populated;
data-model §9 under-specifies the others). Therefore:

- **Recommended:** derive `type`/`owner`/`system`/`domain` for `list` and
  `describe` directly from the resolved `ComponentManifest`s (walk the
  `CatalogSnapshot`'s component set via the Resolver), and the
  `lastExecution*`/`STATUS` columns from `ResolveComponentLatest` /
  `ComponentExecutionIndex`. This needs NO new index axis and unblocks the
  whole PR.
- Do **not** block this PR on populating the empty index axes. If you find a
  command genuinely cannot be built without an axis the spec doesn't
  define (e.g. an O(1) `--owner` filter index), proceed with the manifest-
  walk approach AND file `ai/proposals/task-0038-spec-update.md` proposing
  the axis shape for a later milestone — keep the CLI behavior correct via
  the walk in the meantime, and record the assumption in the report.

`history`: the catalog-local `ComponentExecutionIndex`
(`internal/catalogmodel/indexes.go`) is the read source. If the Resolver
exposes no direct accessor for it, add a small tested read helper (prefer
`internal/catalogstore` via `statestore`, no raw FS) rather than walking
paths in the CLI.

`validate`: re-run `catalogresolve` in strict mode over the selected
source's workspace inputs and surface its `[]ValidationIssue`. Reuse the
same resolve path `refresh` uses (minus the writes) — do not invent a
second resolution entry point.

If you discover any spec under-specification beyond the index axes, proceed
with a documented narrow assumption and file
`ai/proposals/task-0038-spec-update.md` rather than silently diverging.
